# shotAI — Hardening & Cleanup Plan

Implementation plan for **all** structure-review + security-review findings (both run 2026-06-30 via adversarial multi-agent workflows, then a third workflow that produced this dependency-ordered plan and red-teamed the sequencing). Full finding detail lives in the review backlog; this doc is the **execution spec**.

**Threat model:** shotAI is a *local single-user Windows desktop tool*. Primary assets: (1) the Anthropic **API key** (main-process-only), (2) the **redaction guarantee** (blur/crop must be baked into the `flattened` PNG before any send to Claude or any export).

## Conventions
- **Branch off `main`** per track; keep separate from unrelated work.
- **Verification gates (every phase):** `npx tsc --noEmit` + `npm run lint` clean (only the pre-existing `CaptureController` non-null-assertion warning is allowed).
- **Packaging-affecting phases** (P3 sandbox switch, P5 extraResource, P9 rename): also `npm run make`/`npm run package` or at least `npm run build`.
- **Security-critical / runtime-heavy phases:** live `npm start` test of the specific scenario, **plus** direct-call/unit tests where the real UI can't reach the fixed code (see Testing infrastructure below).
- One commit per phase (bisectable); PR granularity per the table.

## Testing infrastructure (prerequisite — "P0.5")
The repo has **no test runner** (`package.json` has no `test` script; no vitest/jest). Several fixes below (S3 merge-invalidation, atomic-write serialization, the extracted pure functions) are **unreachable through the real UI** and need direct-call tests. Add a minimal **vitest** dev-dep + `test` script before P1/P2 so the security-critical fixes are actually verifiable. Pure functions to cover: `confinePath`, `resolveSendableRender`, `applyPatchAndInvalidate`, `parseRect`/`parsePoint`, `capture-geometry` (`unionRect`/`cropToRegion`/`clickBox`/`captureModeFor` golden values), `markerColorFor`.

## Testing on a physical x64 Windows machine

This is **much simpler than the arm64 dev VM** — no Rust/MSVC/Python needed (the `element_locator.dll` is committed; native deps are x64 prebuilts npm installs directly), and none of the `SHOTAI_NO_SANDBOX` / arm64-repair steps apply. Prereqs: **Node 24 LTS + Git**.

```sh
git clone https://github.com/Armadillon44/shotAI.git
cd shotAI && git checkout hardening
npm install        # installs x64 binaries natively; if koffi/get-windows don't load: node scripts/postinstall.mjs
npm start          # real Electron + real main.ts → OS sandbox ON (no env vars) = the shipped default
```

Three levels of testing, easiest first:
1. **`npm start`** (dev) — fastest; verifies the sandbox-on default + lets you run all the live UI checks. (GPU is disabled by default even here — a VM workaround that currently ships to all users; `set SHOTAI_ENABLE_GPU=1 && npm start` to see HW accel. Doesn't affect the sandbox check.)
2. **`npm run package`** then run `.\out\shotAI-win32-x64\shotAI.exe` — the real **packaged** app (asar, `resources/tessdata`, `.ico`, fuses, sandbox) without needing the Squirrel installer. Best single check for S7's packaging path + the exe icon.
3. **`npm run make`** — full Squirrel installer (`out\make\squirrel.windows\x64\*Setup.exe`). Never run end-to-end; `electron-winstaller` will likely need an npm-11 allow-scripts approval first. Do last, for release.

Notes: don't copy `node_modules` from the arm64 VM (install fresh on x64); paste a Claude API key into Settings for the SOP checks.

### ✅ S11 — Packaged app is DOA: `node_modules` not shipped (found + FIXED 2026-06-30, x64) — was a RELEASE BLOCKER
**FIXED:** added a `packageAfterCopy` hook in `forge.config.ts` (`copyProductionNodeModules`) that walks the **actual runtime dependency closure** (each `package.json`'s `dependencies` + `optionalDependencies`, BFS from the root's prod deps) and copies each package dir recursively into the packaged app's `node_modules`. Chose closure-walking over a `package-lock.json` `dev`/`devOptional` filter because npm flags some genuinely-needed transitive deps as `devOptional` (e.g. `readable-stream`, reached via `get-windows → @mapbox/node-pre-gyp → npmlog → are-we-there-yet`), which a flag filter wrongly dropped (caused a second launch crash `Cannot find module 'readable-stream'`). Result: 99 prod packages copied → 153 in the asar incl. the full node-pre-gyp chain; `.node` binaries unpacked via `asar.unpack`. Packaged `shotAI.exe` now launches (`packaged=true`), capture/SOP/offline-OCR all work. Static gates still green (tsc, eslint, 36 vitest). ⚠️ Two iterations were needed (electron-log, then readable-stream); a `npm run make` full-installer smoke test is still worth doing before release. Original diagnosis below.


Running `out\shotAI-win32-x64\shotAI.exe` crashes immediately with an Electron error dialog:
`Uncaught Exception: Error: Cannot find module 'electron-log/main'` (require stack `app.asar\.vite\build\main-*.js`).
**Root cause:** `app.asar` contains only `.vite/` + `package.json` — **zero `node_modules`** (verified via `asar list`; also no `app.asar.unpacked/`). The Vite main/preload builds **externalize** electron-log, the native modules (`uiohook-napi`, `node-screenshots`, `get-windows`, `koffi` + `@koromix/koffi-win32-x64`, `node-screenshots-win32-x64-msvc`), `tesseract.js`, `@anthropic-ai/sdk`, and `zod` (see `vite.main.config.ts`), expecting them at runtime — but `@electron-forge/plugin-vite` packages only the Vite output and does **not** copy `node_modules`, even though the packaged `package.json` declares them under `dependencies`. The app dies on the first external require (`electron-log/main`); every other externalized module (capture, hotkeys, element-locator, OCR, Claude) would fail too. Dev (`npm start`) works only because it runs from the repo's `node_modules`. This is **pre-existing** (not introduced by the hardening phases) and is exactly the gap the "`npm run make` packaging path still untested" note flagged.
**Fix options:** (A) for native/file-based externals (`uiohook-napi`, `node-screenshots`, `get-windows`, `koffi`, `tesseract.js`) keep them external and **copy their `node_modules` (incl. transitive + the optional `*-win32-x64*` binary packages) into the package** via a forge `packageAfterCopy` hook (or fix forge's prod-dep copy); (B) additionally **bundle the pure-JS externals** (`electron-log`, `@anthropic-ai/sdk`, `zod`) into `main.js` by removing them from `vite.main.config.ts` external (bundling the SDK+zod **together** also resolves the dual-zod-instance concern that motivated externalizing them). Re-package and confirm `app.asar` contains the needed `node_modules`, the app launches, capture + SOP + offline OCR all work. ⚠️ Verify on this x64 box — packaging-path-only failures don't show in dev.

UX issues across the editor, SOP pipeline, and project view — distinct from the security/structure hardening above. Several editor items (E1–E3) share a root cause: transient controls/messages are inserted as **block elements in the `ed__main` column above `ed__canvas`**, so each one pushes the image down and squishes the view. The error-banner items (E3, E5) are the same family (`project__error` / editor `error` — persistent, non-dismissable block banners).

- **E1 — Bug: can't edit existing text.** Editing an existing text element opens via `onDblClick` on the `<KText>` node (`Editor.tsx:1137`) → `setTextEntry({ id: a.id, value: a.text, … })`; commit goes through `commitText` (`:585-600`, `update(entry.id, { text, fontSize })`). Double-click isn't reliably reopening the entry — likely the text node only receives pointer events under `tool==='select'` (the `common` props gate interaction), or the dblclick is swallowed. Fix: make editing reachable when a text element is selected (e.g. an explicit "Edit text" affordance in the element header per E2, and/or fix the dblclick listening condition). Verify: add text → save → reopen step → edit the text → save → text updates.
- **E2 — Bug: text add/edit should live in the element header, not a stacked banner.** The text entry renders as a full-width `<div className="ed__textbar">` (`:938-986`) inserted above the canvas, adding yet another banner that squishes the image. Move the add/edit-text field + size + commit into the contextual element header / props area (`ed__topbar` `:786` or `ed__props` `:838`) so it doesn't add a row. (Ties into E1: a header-based edit affordance also fixes the "can't reopen" gap.)
- **E3 — Improvement: OCR "no matches" (and other) errors squish the image.** `autoRedact` sets `setError('No sensitive data detected …')` (`:639`); `error` renders as a block `<p className="project__error">` above the canvas (`:936`), squishing the view. Make it an absolutely-positioned **overlay hovering at the top of `ed__canvas`** with a translucent/opacity background + a **dismiss (×)** button (`setError(null)`). Note `error` is also the sink for save/autoRedact *failures*, so the overlay must read well for both the benign "no matches" notice and real errors (consider distinct styling/severity).
- **E4 — Bug: the SOP "Overview"/intro should not be a discrete step.** `applySopEdits` pushes `plan.intro` as a discrete numbered text step at the top (`sop-apply.ts:64-66`, `makeTextStep(intro.heading, intro.body, aiInserted=true)`; schema `shared/sop.ts:129-130`), so the overview shows up as step [0] in the report/export and is counted/numbered like a real step. It should render as a **non-step preamble/header** above the steps instead. Likely fix: store the intro on the manifest (e.g. `manifest.intro`) rather than as a step, render it as a lead-in in `Report.tsx`/export, and drop it from the numbered-step flow (and from `revertSop`'s step list). Cross-check: `sentSteps = steps.filter(!aiInserted)` (SopPanel:60) and the `!aiInserted` base-rebuild in `applySopEdits` must still behave once the intro is no longer a step. ⚠️ section headings (also `aiInserted` text steps, `:80`) are a separate decision — the user flagged only the **overview/intro**, so don't change section-heading behavior unless asked.
- **E5 — Bug: the Export error persists and isn't dismissable.** `ProjectDetail.tsx:319` renders `{exportErr && <p className="project__error">Export failed: {exportErr}</p>}` — a persistent block banner with no dismiss; it only clears on the next export attempt (`:100`). Same for sibling banners `importErr` (`:318`) and `error` (`:320`). Add a **dismiss (×)** that clears the state (`setExportErr(null)`), and consider auto-clearing on navigation. Same error-banner family as E3 — ideally a shared dismissable-notice component for both the editor and project views. ## Config / behavior backlog (new — 2026-06-30)
- **C1 — GPU enabled by DEFAULT (with a safe fallback).** Requested 2026-06-30. Today `main.ts:96` disables HW acceleration unless `SHOTAI_ENABLE_GPU==='1'` (a VM workaround that currently ships to ALL users → software rendering, which also makes CSS `backdrop-filter` a no-op, see E6). Flip the default to **GPU ON**, and only disable it when we **detect** it won't work — e.g. a virtual machine or an arm64 host running x64 under emulation (the contexts where GPU-context creation aborts startup, as seen by the `Failed to create shared context for virtualization` GPU errors on the dev VM). Keep an env override for manual force on/off. Detection ideas: `app.getGPUFeatureStatus()` / `app.getGPUInfo()` after `ready`, CPU arch vs. process arch mismatch (arm64 host + x64 emulation), or a known-VM heuristic; fall back to `disableHardwareAcceleration()` + the GPU switches only then. Verify on real x64 HW (GPU on) AND the arm64 dev VM (must still launch). Bonus: with GPU on, the E6 notice `backdrop-filter` blur starts working.
